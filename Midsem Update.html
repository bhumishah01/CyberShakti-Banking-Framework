# Midsem Update – RuralShield (Cybersecurity Framework for Rural Digital Banking)

## Problem Statement (Official)
Problem Statement ID: 25205  
Title: Cybersecurity Framework for Rural Digital Banking  
Description: Develop a lightweight cybersecurity framework to secure digital banking transactions for rural users, focusing on fraud detection and user authentication. The solution should be compatible with low-end smartphones and limited internet connectivity.  
Expected Outcome: A software prototype reducing fraud incidents by 20% in rural banking, with a simple interface for users to authenticate transactions securely.  
Technical Feasibility: Utilizes open-source encryption libraries and machine learning for anomaly detection, optimized for low-resource devices.  
Organization: Government of Odisha  
Department: E & IT Department  
Category/Theme: Software — Blockchain & Cybersecurity


## 1) Core Idea (Simple)
RuralShield is an offline‑first security system for rural digital banking. It protects users by:
- storing data safely on the phone first (SQLite),
- detecting fraud locally,
- requiring extra verification for risky transactions, and
- syncing later when internet is available.

This matches rural realities: low‑end phones, weak connectivity, and higher fraud risk.

## 3) What We Planned
We designed a small system with:
- Local encrypted storage (SQLite)
- Fraud scoring rules for anomaly detection
- Secure transaction flow with hold/block actions
- Trusted contact approvals for risky transactions
- Audit log & change log for transparency
- Offline outbox + later sync
- Simple web UI for demo and understanding
- Multilingual UI for rural users

## 4) What We Built (Everything Visible on Localhost)
### A) Dashboard (Main Page)
- Shows counts for users, transactions, pending sync, synced, held, blocked, high‑risk, audit events.
- All data is read from the local SQLite database.
- This page is controlled by:
  - `src/ui/app.py`
  - `src/ui/templates/index.html`
  - `src/ui/static/styles.css`

### B) Register / Replace User
- Adds a new user locally with PIN‑based authentication.
- PIN is hashed using scrypt (slow hash for security).
- User record is written in SQLite.
- File responsible:
  - `src/auth/service.py` (`create_user`)

### C) Create Secure Transaction
- Takes user ID, PIN, amount, recipient.
- Transaction is encrypted locally before saving.
- Fraud rules run locally (no internet).
- If safe: transaction goes into outbox.
- If risky: transaction is held or blocked.
- File responsible:
  - `src/database/transaction_store.py` (`create_secure_transaction`)

### D) Fraud Risk Detection
- Fraud logic is rule‑based (amount, odd hour, new recipient, rapid burst, failed logins).
- Gives a risk score + reasons.
- File responsible:
  - `src/fraud/engine.py`

### E) Trusted Contact & Panic Freeze
- Users can set a trusted contact for approvals.
- Panic freeze blocks outgoing transfers for a set time.
- Stored in user `auth_config` in SQLite.
- File responsible:
  - `src/auth/service.py`

### F) Release Held Transaction (Step‑Up Verification)
- Held transactions can be released if trusted approval code is valid.
- This demonstrates extra authentication for high‑risk transfers.
- File responsible:
  - `src/database/transaction_store.py` (`release_held_transaction`)

### G) Transaction List Page
- Shows decrypted transaction details after PIN authentication.
- If integrity check fails, transaction is shown as rejected.
- File responsible:
  - `src/database/transaction_store.py` (`list_secure_transactions`)
  - `src/ui/templates/transactions.html`

### H) Sync and Offline Queue
- Transactions are first placed into the outbox queue (offline first).
- Sync can be done later (manual or simulated).
- Outbox status: PENDING, RETRYING, SYNCED, HOLD, BLOCKED.
- File responsible:
  - `src/sync/manager.py`
  - `src/sync/client.py`

### I) NEW: Sync Queue Page
- A dedicated UI page that shows the local outbox queue.
- Shows which transactions are waiting to sync.
- A Simulate Night Sync button moves pending to synced for demo.
- File responsible:
  - `src/ui/app.py` (`/sync/queue`, `/sync/simulate`)
  - `src/ui/templates/sync_queue.html`

### J) Audit Chain
- Every important action is stored in an append‑only audit log.
- Shows tamper‑evidence (if data is edited, chain breaks).
- File responsible:
  - `src/audit/chain.py`

### K) Change Log (Old/New Values)
- Tracks edits locally (e.g., user replace, transaction status change).
- Can be exported to CSV for Excel.
- File responsible:
  - `src/audit/change_log.py`

### L) Fraud Impact Report
- Shows simulator results for fraud controls.
- Demonstrates how many risky transactions are held/blocked.
- File responsible:
  - `src/ui/templates/impact_report.html`
  - `src/ui/app.py`

### M) Demo Walkthrough (Professor Mode)
- A scripted walkthrough page for explaining the demo.
- Shows a deterministic demo scenario (low‑risk + high‑risk + freeze).
- File responsible:
  - `src/ui/templates/demo_walkthrough.html`

### N) Audio Guidance (Voice)
- When risk is detected, a voice button appears.
- The app uses browser speech synthesis to read safety guidance aloud.
- No internet is required.
- File responsible:
  - `src/ui/templates/index.html` (JS speechSynthesis)
  - `src/ui/templates/agent.html` (same)

### O) Multi‑Language UI
- Supports English, Hindi, Odia, Gujarati, German.
- All visible UI text switches based on language selection.
- File responsible:
  - `src/ui/app.py` (i18n bundles)
  - templates use `{{ i18n.key }}`

## 5) Tech Stack Used (Simple)
- Python 3 for all backend logic
- FastAPI for the local web server
- SQLite for offline‑first storage
- Jinja2 templates for HTML pages
- Cryptography library for encryption/signature
- Browser SpeechSynthesis API for audio prompts

## 6) How We Implemented (Simple Steps)
1. Local DB created with tables: users, transactions, outbox, audit_log, change_log.
2. Users created with hashed PINs and settings stored locally.
3. Transactions created: encrypted, scored for fraud, then queued for sync.
4. Fraud decisions: allow, step‑up, hold, or block.
5. Trusted contact used for approvals in risky cases.
6. Audit & change logs record every critical action.
7. UI built to show all flows in one demo‑friendly portal.
8. Offline queue page added to show pending sync + simulate night sync.

## 7) Languages and Tools Used
